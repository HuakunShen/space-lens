# Repository Guidelines

## Project Structure & Module Organization

`space-lens` is a NAPI-RS package that exposes Rust disk-usage logic to Node.js. The native exports live in `src/lib.rs`, with the build hook in `build.rs`. Root-level `index.js` and `index.d.ts` are generated by NAPI-RS and provide the JavaScript loader and TypeScript declarations. Tests live in `__test__/index.spec.mjs` and use AVA. Platform-specific npm package metadata is under `npm/<target>/`. The vendored `dust` reference source and fixtures are in `vendors/dust/`; avoid editing it unless intentionally updating the embedded dependency behavior.

## Build, Test, and Development Commands

- `yarn install`: install JavaScript tooling from `yarn.lock`.
- `yarn build`: build the release native addon for the current platform.
- `yarn build:debug`: build a debug native addon for local iteration.
- `yarn test`: run the AVA test suite in `__test__/`.
- `cargo fmt`: format Rust code using `rustfmt.toml`.
- `cargo clippy --all-targets --all-features`: lint Rust; `src/lib.rs` denies Clippy warnings.
- `yarn artifacts`, `yarn universal`, `yarn version`: NAPI-RS release helpers for generated artifacts and package versions.

## Coding Style & Naming Conventions

Rust uses edition 2021 and two-space indentation from `rustfmt.toml`. Keep exported NAPI structs and functions in `src/lib.rs` small and explicit, and use `#[napi(js_name = "...")]` when Rust snake_case fields need JavaScript camelCase names, such as `ignoreHidden` or `fullPath`. JavaScript tests and examples use ESM imports, double quotes, and camelCase API names.

## Testing Guidelines

Add or update AVA tests in `__test__/*.spec.mjs` for public API changes. Build the native addon before running tests when Rust exports change. Prefer tests that assert returned shapes and key sizes rather than relying only on console output. For filesystem-sensitive behavior, use deterministic fixture directories when possible.

## Commit & Pull Request Guidelines

Recent history uses short imperative or release-style subjects, for example `Update README to include usage examples...`, `Refactor buildDirectoryTree...`, and version tags like `0.0.6`. Keep commits focused and mention the affected API when relevant. Pull requests should include a concise summary, test results (`yarn test`, `cargo fmt`, `cargo clippy` as applicable), linked issues, and usage or output examples for public API changes.

## Security & Configuration Tips

Do not commit `.env`, `*.node`, `target/`, or generated package archives. Treat platform binaries as build artifacts unless a release process explicitly requires them.
