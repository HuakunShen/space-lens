## Core Decision/Topic

Add a compact directory scanner and a real benchmark/export CLI for evaluating large-folder scans without retaining every ignored descendant in memory.

## Options Considered

- Keep using the dust-backed full tree API only. This preserved existing behavior but produced very large trees for folders with many small files.
- Modify the vendored dust source to respect `.gitignore` while walking. This would have reused dust internals but made vendor maintenance and publishing riskier.
- Add a first-party compact scanner in this repository. This keeps vendor code untouched and lets the product API control how ignored directories are represented.

## Final Decision & Rationale

Implement a first-party `scanCompact` API and keep `du-dust` temporarily as a compatibility layer and benchmark baseline. The compact scanner respects `.gitignore`, can summarize ignored directories as collapsed leaf nodes, and avoids storing unnecessary descendants. Dust remains useful for `buildDirectoryTree`, `getLargestNodes`, and performance/correctness comparisons while the compact scanner matures.

## Key Changes Made

- Added `src/scanner.rs` with parallel directory traversal, `.gitignore` stack handling, collapsed ignored directory summaries, allocated-size accounting, and hard-link deduplication by inode/device on Unix.
- Exposed `scanCompact` through napi and regenerated JS/TypeScript bindings.
- Added AVA coverage for gitignored directory summarize/exclude behavior and hard-link deduplication.
- Added a Citty-based benchmark CLI in `cli.ts` and `scripts/benchmark-cli.ts` with modes for `dust`, `compact`, and `both`, plus JSON tree export.
- Added TypeScript configuration and `yarn typecheck` for the local CLI.
- Updated README with API usage, compact scanner behavior, benchmark CLI commands, and development commands.

## Future Considerations

- Remove `du-dust` once compact scanning no longer needs dust as a public compatibility API or benchmark baseline.
- Add a compact-tree top-N helper so CLI comparisons do not depend on dust for ranking.
- Consider a persistent snapshot/query layer if scans of much larger disks push compact tree memory beyond acceptable limits.
